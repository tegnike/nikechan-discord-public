# 広域チャンネル分析メモ（discord-summary）

このメモは、`discord-summary` を「短い要約」ではなく「運用分析」用途で回すときの標準メモ。

## 目的
- リクエストが「もっと広く」「分析して」「傾向を教えて」「次のアクションが欲しい」系の場合に、
  120件レベルの軽量要約ではなく、定量＋定性で再利用しやすく圧縮する。

## 推奨取得戦略（可逆）
1. **比較基準を先に取る（可能なら）**
   - `discord-summary`利用時と比較しやすいよう、最初に120件取得を試みる。
   - 直近運用の簡易指標との対比が取りやすい。
2. **要件に応じて拡張取得**
   - `--limit 500` を基本拡張値として再取得。
   - さらに不足する/境界が不明瞭なら `--limit 1000` まで拡張。
3. **二次要約より生JSON優先**
   - `discord-summary`やチャット上の省略出力は要約・圧縮されるため、最終集計は
     `discord-history fetch` の `messages` 本体（`author/content/timestamp/embeds/jump_url`）を直接読んで行う。

## 集計テンプレ（最低限）
- 総件数 / bot比 / human比
- トップ投稿者（表示名優先、無ければusername）
- 日次・時間帯（必要なら4hバケット）
- カテゴリ重複ヒット（検索・リマインダー・動画・論文・コスト・雑談）
- 人間依頼文（要約・まとめ等）の件数と最古/最新
- 長い沈黙（例: 6時間超/12時間超）のギャップ抽出
- 連続投稿（bot連投や同報告の反復）検出

## 出力ガイド
- 依頼が「運用レビュー」寄りなら、**全体俯瞰→定量→話題フロー→未決/次アクション**の順で簡潔提示。
- 定量が多い場合は、数値だけの羅列にならないよう「なぜこの傾向か」を1〜2行添える。
- もし再現可能性が必要なら、最後に取得条件（`from/to/limit/channel`）を明記。

## 落とし穴（実運用で注意）
- 同種メッセージ（cron失敗連打など）があるとテーマ比率が歪む。
  → 初見では`同種ログの塊`を分離して「運用ノイズ」として注記すると読みやすい。
- bot投稿が多い帯だけ見てユーザー要件を判断しない。
  → `human_request`相当のヒント（要約依頼・質問・意思決定要求）を別抽出すると次アクション提案が安定。
- `--limit` が大きくなると、会話構造が「短期イベント」から「複数テーマ併存」に変わる。
  → 単独の“平均値”より、日次分割で比較して傾向変化を示す。
